## What This Document Is

This document explains how VIGIL receives, surfaces, and supports human response to
security alerts generated by the SOVEREIGN Security Observability Framework. It is
written for LENS — the platform's orientation and explanation module — so that LENS
can answer user questions about alert response accurately and in plain language.

---

## The Alert Pipeline — How an Alert Reaches VIGIL

The SOVEREIGN Security Observability Framework runs continuously beneath every product
in the platform. It watches for anomalies — unusual patterns in Logger events, honeytoken
activations, and behavioral drift — and when it detects one, it generates an alert.

That alert travels to VIGIL's Alert Queue. VIGIL is the only user-facing surface where
Security Framework alerts appear. No product handles its own alerts. No product builds
its own anomaly detection. The Security Framework is the platform's shared nervous
system, and VIGIL is where a human sees and responds to what it finds.

The path looks like this:

```
Security Framework detects anomaly
        ↓
Alert generated with severity, source product, event type, and timestamp
        ↓
Alert routed to VIGIL_ALERT_ENDPOINT
        ↓
VIGIL Alert Queue displays the alert to the authorized operator
        ↓
Anomaly Triage Assistant analyzes the alert (AI-assisted, human-decided)
        ↓
Human operator reviews the analysis and decides on a response action
        ↓
Response action recorded with HUMAN_DECISION event (decision_type, actor, workflow_step_id)
```

## Who Can See Alerts

The VIGIL module is access-controlled at the platform's highest tier. Only users with
the `PLATFORM_ADMIN` or `SYSTEM_ADMIN` role can mount VIGIL. The system enforces this
structurally — it is not a UI restriction that can be navigated around. Attempting to
access VIGIL with any other role produces an access-denied error before the module loads.

This is by design. Security alerts contain sensitive operational information about
platform behavior. Restricting access to platform and system administrators ensures
that alert data does not reach users who lack the authority or context to act on it.

## What the Alert Queue Shows

Each alert in the Queue displays:

- **Severity** — the priority level assigned by the Security Framework (P1 through P4,
  where P1 is the most critical)
- **Source product** — which SOVEREIGN product the anomaly originated in
- **Event type** — the category of anomalous behavior detected
- **Timestamp** — when the Security Framework detected the anomaly
- **Status** — whether the alert is new, under review, or resolved

CPMI alerts receive special handling. Because CPMI's governance outputs flow to every
other product simultaneously, CPMI operates under an enhanced monitoring tier with a
tighter anomaly threshold (0.7× the standard). A `CPMI_DRIFT_DETECTED` alert in the
Queue signals a potential platform-wide governance concern, not a single-product issue.

## The Anomaly Triage Assistant

When an operator opens an alert, the Anomaly Triage Assistant — an AI agent running
under `vigil-triage-analyst` — analyzes the alert and produces a structured triage
recommendation. The recommendation includes:

- A plain-language explanation of what the anomaly means
- The likely source and scope of the issue
- Suggested response actions, ranked by priority
- Any related alerts that may indicate a pattern

The Triage Assistant's output is a recommendation, not a decision. The human operator
reviews it, may request additional analysis, and then decides on the response action.
The Assistant cannot close an alert, escalate an alert, or take any action on the
platform. Only the human operator can do those things.

This boundary is not a technical limitation — it is a governance requirement. Every
alert response is a human decision, recorded as a `HUMAN_DECISION` Logger event with
the operator's identity, the decision type, and a workflow step ID that ties the
response to the original alert.

## What Happens When the Alert Endpoint Is Not Configured

VIGIL requires a `VIGIL_ALERT_ENDPOINT` to receive live alerts from the Security
Framework. If this endpoint is not configured — as is the case during development and
in environments where the Security Framework has not yet been wired — the Alert Queue
displays a configuration notice.

The notice states clearly that the endpoint is absent and that alerts are not flowing.
It does not generate an error or crash the module. An empty queue and an unconfigured
endpoint are treated differently: an empty queue means no alerts are currently active;
an unconfigured endpoint means the Alert Queue is not yet operational.

The distinction matters because an operator seeing an empty queue should not conclude
that the platform is secure — they should first confirm that the endpoint is configured
and alerts are actually flowing.

## Alert Severity and Response Expectations

| Severity | Description | Expected Response Time |
|---|---|---|
| P1 | Critical — potential active breach or platform-wide governance failure | Immediate |
| P2 | High — significant anomaly requiring prompt investigation | Within the hour |
| P3 | Medium — anomaly requiring investigation at next opportunity | Within the day |
| P4 | Low — informational; pattern to monitor | Next review cycle |

P1 and P2 alerts are routed without batching — they appear in the Alert Queue
immediately. P3 and P4 alerts may be batched depending on Security Framework
configuration. CPMI alerts are always routed at P1/P2 priority regardless of the
underlying severity classification, because of their platform-wide dependency status.

## Key Facts for LENS to Surface

- VIGIL is the only place in the platform where Security Framework alerts appear
- Only PLATFORM_ADMIN and SYSTEM_ADMIN roles can access VIGIL
- The Anomaly Triage Assistant recommends — the human operator decides
- Every alert response is recorded as a HUMAN_DECISION event
- An empty Alert Queue is not the same as a configured-and-quiet queue
- CPMI alerts are always treated at elevated priority
- The Security Framework runs beneath all six primary products — alerts can originate
  from any of them
